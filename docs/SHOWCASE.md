# 顺发 · 案例研究

> 一个人，用 AI 从 0 到可上线，做完一个完整产品的全过程。
>
> 这篇文档不是「怎么跑起来」（那是 [README](../README.md)），而是「为什么这么做、做对了什么、哪里诚实地承认做得不够」。
> 它想展示三件事：**Vibe Coding 的工程能力**、**做产品的思路**、**内容运营的思路**。

---

## 0. 一句话定位

顺发是一个**能力展示项目**，不是一门生意。它解决一个真实存在的小问题——「想写，但总觉得写得不够好，于是永远不发布」——并把这个问题做成一个**闭环可度量、可上线、可复盘**的完整产品。

衡量它的不是 DAU 或付费转化，而是四个能力维度：

| 维度 | 想证明的事 | 本仓库里的证据 |
|------|-----------|---------------|
| **Web Coding / 可落地** | 一个人能用 AI 端到端交付并部署 | FastAPI + Next.js 16 + 小程序三端、16 个 Alembic 迁移、293 个测试函数、4 条 CI 流水线、上线检查清单 |
| **产品迭代意识** | 不是一次性堆功能，而是假设驱动地演进 | 闭环 PRD、4 个核心假设表、状态机、按需重构的迁移历史 |
| **运营能力** | 留存靠度量而非感觉 | 已落地的埋点 / 漏斗 / 北极星指标（`analytics_queries.py`） |
| **产品思考** | 敢于写「什么不是优势 + 下一步往哪走」 | 把游戏化定义为「复盘内容」而非信仰、减法实验 |

## 1. 问题与洞察（产品思路）

### 1.1 真问题

科技从业者大多有表达意愿，但卡在同一处：**完美主义**。写到一半觉得不够好，于是不发；攒了几个想法，永远停在草稿箱。

顺发的产品主张只有一句：**「写完了再想好不好。」** 把「开始写」这件需要意志力的事，变成不需要意志力的日常动作。

### 1.2 三步闭环

```
AI 给选题  →  一对一讨论打磨角度  →  生成初稿一键发布
```

对应到状态机，每一次发文都是一条可观测的状态流：

```
topic_selected → discussing → draft_ready → pending → completed
```

把「写作」拆成有限状态，不只是工程便利——它让产品的每一步都**可埋点、可度量、可干预**。这是后面运营章节的地基。

### 1.3 假设驱动，而不是信仰驱动

做这个产品时，我没有假设「用户一定会爱上打卡」。相反，我把核心信念写成**可证伪的假设**，并为每条假设预留度量手段：

| 假设 | 如果为假会怎样 | 怎么度量 |
|------|---------------|---------|
| 用户卡在「不敢发」而非「不会写」 | 选题/讨论环节没人卡，卡在发布前 | 漏斗各步转化率 |
| AI 讨论能降低开始写的门槛 | 讨论轮次集中在 0 轮、直接放弃 | `discuss_round` 事件分布 |
| 连胜机制能驱动 ≥3 天留存 | 北极星比例长期低位 | ≥3 天连胜占发布者比例 |
| 一个人用 AI 能交付可上线质量 | 上线检查清单过不了 | launch-checklist + smoke 测试 |

> 这张表是整份产品观的核心：**先承认自己可能错，再用数据来纠错。**

## 2. Vibe Coding：一个人的全栈交付

### 2.1 交付物规模

| 项 | 内容 |
|----|------|
| 后端 | FastAPI + SQLAlchemy（同步 Session + SQLite WAL），约 20 个 service 模块 |
| 前端 | Next.js 16 App Router + Tailwind（login / topics / discuss / preview / compose / drafts / history / profile / settings / image-cards） |
| 小程序 | 微信端 8 个页面（index / topics / discuss / preview / drafts / history / profile / settings） |
| 数据模型 | 7 张表：User、CheckIn、TopicHistory、Achievement、HotTopic、Event |
| 数据库演进 | **16 个 Alembic 迁移**，按需求逐步演进而非一次成型 |
| 测试 | **293 个测试函数**（359 passed，含参数化用例），覆盖积分、连胜、状态机、漏斗、迁移、上线 smoke |
| CI | **4 条流水线**：backend-test、web-test、launch-smoke、rss_cron |

### 2.2 工程上做对的几个决定

这些不是「能跑就行」，而是**有意识地为可维护性和可上线性付出的成本**：

- **BYOK（用户自带 Key）+ Fernet 加密。** AI 费用不挂在作者头上，密钥加密存储、作者看不到明文。`get_resolved_api_key` 做三级解析（用户 Key → 系统 fallback → 拒绝），既能开源演示又能强制线上自付费。
- **埋点是「尽力而为」的旁路。** `analytics.track` 的失败**绝不能**打断主请求流——度量重要，但不能让度量拖垮产品。
- **迁移历史会说话。** 从 `initial_schema` 到 `add_content_feedback_fields`、`add_token_version_to_users`、`add_hot_topics_table`……每个迁移对应一次真实的产品演进，而不是把 schema 推倒重来。这本身就是「迭代意识」的代码证据。
- **上线不是终点而是清单。** [docs/launch-checklist.md](launch-checklist.md) 把迁移回滚、PostgreSQL 验证、BYOK 脱敏、超时策略、限流、Web 构建、人工 smoke 全部列成可勾选项；`test_launch_smoke.py` 把其中一部分变成自动化。

### 2.3 AI 协作方式

整个产品用「人定方向、AI 写实现」的方式推进：我负责定义问题、拆状态机、写假设、定埋点口径；AI 负责把 service、迁移、测试落地。关键纪律是——**每一个 AI 写的模块都配测试**（293 个测试函数就是这条纪律的产物），这样 Vibe Coding 才不会变成「跑起来但不敢动」的代码。

## 3. 内容运营思路（产品内的运营设计）

顺发把「内容运营」直接做进了产品里，而不是当成发出去之后才考虑的事。运营在这里有四个抓手：

### 3.1 选题供给：热点发现管线

写作最大的摩擦是「今天写什么」。顺发用一条**自动化热点管线**解决供给：

```
RSS 抓取（rss_cron 流水线） → 事实增强（fact_enrichment_service）
        → 网络检索补料（web_search） → 每日 3 条推荐热点（/api/hot_topics/today）
```

- `rss_service` + `rss_cron.yml` 定时抓取信息源；
- `fact_enrichment_service` 给热点补充事实，避免 AI 凭空编造（这条来自一条明确的内容红线：**不得编造用户没提到的例子**）；
- 用户每天打开就有 3 个「有料的角度」可选，把选题从一道难题变成一次挑选。

这是运营里最关键的一环：**降低供给摩擦，等于降低开始写的摩擦。**

### 3.2 发文节奏：连胜 = 内容日历

连胜（streak）表面是游戏化，本质是**内容日历习惯**。它把「每天产出一条」这件运营纪律，包装成用户自己愿意维护的连续记录。配套的积分/等级/钻石只是把这条日历的反馈做得更即时：

| 指标 | 规则 | 运营意图 |
|------|------|---------|
| 每日积分 | 基础 +30，连续加成 +5/天（cap +30），选题 +10，讨论 +3/轮（cap +9），按时 +5 | 奖励「持续」与「认真打磨」，而非一次性爆发 |
| 等级 | 7 级：0/100/300/700/1500/3100/6300 | 给长期坚持一个可见的进度条 |
| 连胜 | 连续发文天数，断更超 1 天清零 | 制造「不想断」的轻微压力 = 内容日历的钩子 |

### 3.3 度量：埋点 → 漏斗 → 北极星（已落地）

运营不能靠感觉。顺发把整条留存逻辑做成了**可计算的指标**，代码在 `backend/app/services/analytics_queries.py`：

**漏斗**（按去重用户数，重试不灌水）：

```
注册 → 配置 Key → 选题 → 讨论 → 初稿 → 发布
register → key_configured → topic_selected → discuss_round → draft_generated → publish
```

每一步都给出「相对首步转化」和「相对上一步转化」，哪一环漏水一目了然。

**北极星指标**（诚实地设计成能打自己脸的口径）：

```
北极星 = 连胜 ≥ 3 天的用户数 / 曾经发布过的用户数
```

> 选这个口径是有意为之：**如果连胜机制根本没起作用，这个比例就会一直很低——这是最强的「游戏化没用」的信号。**
> 我没有把指标设计成只会变好看的样子，而是让它有能力告诉我「这套机制该被砍掉」。

### 3.4 运营的反馈回路

供给（热点）→ 节奏（连胜/日历）→ 度量（漏斗/北极星）→ 回到供给与机制调整。这条回路让运营从「凭感觉发」变成「看漏斗哪一步漏、就改哪一步」。

## 4. 诚实的复盘（什么不是优势）

能力展示最容易翻车的地方，是把所有东西都说成优势。这一节专门写**不是优势的部分**和**下一步**。

### 4.1 游戏化是「复盘内容」，不是信仰

我**不相信**积分/等级/钻石本身能留住人。它们在这个项目里的真正角色，是**一个可被证伪的实验**：北极星比例（§3.3）就是用来判它生死的裁判。如果数据说连胜没驱动 ≥3 天留存，那么正确的产品动作是——**做减法，把它砍掉**，而不是再加一层奖励。

这就是「减法实验」：把「敢于移除自己做的功能」当成一种复盘产出，而不是一个商业决策。

### 4.2 当前明确的短板

- **同步 Session 跑在 async FastAPI 上。** SQLite WAL 下可接受，但不是生产姿态；要扛并发得上 AsyncSession / PostgreSQL。
- **JWT 无刷新 / 无吊销。** MVP 够用，但不是安全终态（已加 `token_version` 留了吊销的口子）。
- **真实用户数据尚少。** 漏斗和北极星的口径已就位，但只有跑起真实流量后，§1.3 那四个假设才能真正被验证或证伪。
- **提醒靠 App 内轮询。** 真·推送需要微信订阅消息，审批复杂，暂缓。

### 4.3 下一步往哪走

1. 跑真实流量，让漏斗和北极星从「能算」变成「有结论」；
2. 按北极星结果对游戏化做加法或减法（敢砍）；
3. 把存储从 SQLite/同步 Session 推进到 PostgreSQL/AsyncSession；
4. 用漏斗找出转化最低的一步，针对性优化（大概率是「配置 Key」或「首次讨论」）。
